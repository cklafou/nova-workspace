## FINDINGS FROM CHAT LOG REVIEW

### What actually happened during the overnight/morning sessions

From reading `logs/chat_sessions/2026-05-08_17-44-22_chat.jsonl` (610 messages):

1. **Nova's text was routing to Thoughts pane instead of chat.** When she used tool calls in autonomous mode, her actual summary text ended up in the Thoughts sidebar rather than chat. Moving panes triggered a UI refresh that unstuck it — a symptom of a streaming state desync where thinking tokens and response tokens weren't being assigned to the right DOM target.

2. **Autonomy loop kept firing on the same message.** During today's test, autonomous heartbeats kept re-processing the original "what's 1+1" message, generating ~11 repetitive responses. This is because the heartbeat injects the full conversation context and Nova doesn't have a way to know she already responded. The heartbeat needs to clearly mark processed messages or maintain a "last handled" pointer.

3. **Nova is NOT actually using the Tasking system during autonomous sessions.** Logs show 60 heartbeat ticks in a row where Nova never opened `Thoughts/priority.md`, never created a task folder, and never moved items. She just answered messages in chat. The HEARTBEAT.md instructions exist, but they're not being enforced mechanically — they depend on Nova deciding to follow them.

4. **Tools tab was blank throughout all sessions.** Nova's actual tool calls (read_file, write_file visible in log) never populated the Tools panel. Confirmed from code: `tool_executed` WS events are referenced in the client but are NEVER broadcast by server.py. Zero code fires them.

- **UI "Thoughts" pane** = live reasoning (the streaming reasoning block, extended thinking)

This causes constant confusion. When Cole says "thoughts pane", it's ambiguous. When Nova reads her instructions, "Thoughts" means her task queue, but the UI pane called Thoughts shows her reasoning. They clash badly.

## ISSUE 2 — Tools Tab: Wire It Up (Currently Completely Dead)

### What's happening
The Tools tab shows "No tool calls yet. Executions stream in live." and never changes — for any session, ever.

The client listens for two WS event types:
- `tool_executed` — detailed tool call result (name, input, result, error)
- `nova_activity` — Nova's bridge actions (`[WRITE:]`, `[READ:]`, `[EXEC:]`)

The problem: **`tool_executed` is never broadcast anywhere in server.py.** The event type is wired in the client handler but zero server code ever sends it.

`nova_activity` IS emitted by server.py, but only when Nova uses the old bracket-syntax bridge (`[WRITE: path]` etc.). When Nova uses the Anthropic API's native tool_use (the `read_file`, `write_file`, `list_dir` tool calls visible throughout the logs), those go through the orchestrator/client loop and the Tools tab never hears about them.

### What we'd fix
**In `server.py`:** After each tool_use block completes (in the streaming loop where we already accumulate `full` response), emit a `tool_executed` broadcast with:
```python
await broadcast({
    "type": "tool_executed",
    "tool": tool_name,        # e.g. "read_file"
    "input": tool_input,      # dict of params
    "result": result_text,    # truncated result
    "error": None,
    "duration_ms": elapsed
})
```

This requires hooking into the Anthropic streaming loop where `content_block_start` / `content_block_delta` / `content_block_stop` events come in for `type: tool_use` and `type: tool_result`.

**In `index.html`:** The `addTool(d)` function already exists and renders correctly. No client changes needed — it just needs events to arrive.

## ISSUE 3 — Thoughts Pane: Session Reasoning Log

### What's happening
The "LAST REASONING" block only shows the *most recent* reasoning. Once Nova generates another response, the previous reasoning is gone. You can't scroll back through what Nova was thinking across the session.

### What we'd fix
Add a **"Session Log ▶"** collapsible section at the bottom of the Thoughts panel (below Browse Tasking), similar to the Tasking browser. When expanded, it shows a scrollable list of every reasoning block from the current session, newest-first:

```
▼ Session Log (8 entries)
  [07:24] "Cole is asking me a simple math question..."   [view]
  [07:22] "Cole wants to verify the Tasking system..."   [view]
  [07:20] "Cole is testing autonomous mode..."           [view]
  ...
```

Each entry shows the timestamp and first ~60 chars. Clicking "view" expands the full reasoning inline (or opens the same modal overlay as the Tasking browser uses).

**Implementation:** The client already stores reasoning in `_liveThinkBuf`. We add a `_thinkHistory` array — on each `liveThinkEnd()`, push `{ts: Date.now(), text: _liveThinkBuf}` to the history (cap at 50 entries). The Session Log section renders from this array.

**What files change:** `static/index.html` only — JS array + rendering function, no server changes needed.

## ISSUE 4 — Queue System: Make Tasking Real

### The problem
`Tasking/priority.md` is a markdown file that Nova is supposed to read and write manually every heartbeat. In practice she never does. Even when she does (as during the morning test), the queue has no enforcement — she can add tasks to it and then never execute them, or add duplicate tasks, or forget what she already did.

The current "queue" has no:
- Server awareness (server.py doesn't read it)
- UI for Cole to add/view/manage tasks
- Auto-injection into Nova's context each heartbeat
- Completion tracking beyond Nova writing markdown

### What we'd build

**Layer 1 — Server-side queue file (`memory/task_queue.json`):**
A structured JSON file the server manages, parallel to the existing markdown system:
```json
{
  "tasks": [
    {
      "id": "uuid",
      "title": "Check GPU memory leak",
      "priority": 2,
      "status": "active",
      "created": 1234567890,
      "folder": "Tasking/GPU_LEAK_CHECK/",
      "notes": "Cole noticed VRAM growing"
    }
  ],
  "last_updated": 1234567890
}
```

**Layer 2 — REST API endpoints (server.py):**
- `GET /api/queue` — return current tasks
- `POST /api/queue/add` — add task `{title, priority, notes}`
- `POST /api/queue/complete` — mark task done `{id}`
- `POST /api/queue/cancel` — cancel task `{id}`

**Layer 3 — UI in the Tools tab (repurpose it):**
The Tools tab is effectively dead right now (see Issue 2). Once tools are wired up, the tab has two halves:
- **Top:** Queue panel — shows active tasks from `task_queue.json` with Add Task button and completion controls
- **Bottom:** Tool call feed — live tool execution log (from Issue 2 fix)

Queue UI has:
- "＋ Add Task" button → inline form: title, priority (P1–P4), notes
- Each task row shows: priority badge, title, "✓ Done" and "✗ Cancel" buttons
- Tasks auto-sort by priority
- "Done" tasks move to a collapsed "Completed" section

**Layer 4 — Heartbeat auto-injection (server.py):**
On every autonomous heartbeat, before sending Nova's context, server.py reads `task_queue.json` and prepends a formatted summary:
```
[TASK QUEUE — 3 active tasks]
P1: Check GPU memory leak (folder: Tasking/GPU_LEAK_CHECK/)
P2: Write session summary for today
P4: Clean old log files
Read Tasking/[folder]/master.md before starting any task.
```
This gives Nova the queue state without her having to remember to read the file herself.

## ISSUE 5 — Autonomous Loop: Stop Re-Processing Old Messages

### The problem
During today's test, one "what's 1+1" message caused ~11 autonomous heartbeats all generating answers. The heartbeat system doesn't have a mechanism to say "I already handled this message — if nothing is new, execute tasks from the queue instead of re-answering."

### What we'd fix
In `server.py`'s heartbeat injection, track the last message Cole sent (by ID or content hash) and compare on each heartbeat:
- **If Cole sent a new message since last heartbeat:** Include the full conversation context + respond to it
- **If nothing new from Cole:** Skip the chat context, inject only the task queue summary and instruct Nova to execute her next queued task (not answer old messages)

This prevents the "keep answering the same question 11 times" loop and forces autonomous heartbeats toward actual work.

**What files change:** `server.py` — ~20 lines in the heartbeat dispatch logic, tracking `_last_cole_msg_id`.
