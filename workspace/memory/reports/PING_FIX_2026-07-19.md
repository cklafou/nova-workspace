# The ping that never left the machine

_2026-07-19, Fable. Cole: "she just failed to ping once because of an emoji breaking it, and
couldn't ping again because of the ping guard."_

Two bugs, both mine, and the second one turned the first into a silence.

---

## Bug 1 — one em-dash, and the script never ran. Not once.

Her first-ever ping came back with this:

    At ...\ping_claude.ps1:78 char:43
    +   $text = "[Nova is pinging you �?" she reached out on her own, ...

Line 78 was mine:

    $text = "[Nova is pinging you — she reached out on her own, unprompted]`n" + $text

**Windows PowerShell 5.1 reads a `.ps1` with no UTF-8 BOM using the ANSI codepage (cp1252).**
An em-dash is UTF-8 `E2 80 94`. Under cp1252 those three bytes are three characters — `â`, `€`,
and `0x94`, which is **`”`, a right curly double quote.** PowerShell accepts curly quotes as real
string delimiters. So the string closed in the middle of the sentence and the file failed to
**parse**.

A parse error is compile-time. Nothing executed — not the logger, not the queue, not the `catch`.
That is why `logs/ping_claude.log` did not exist: **the script had never successfully run since
the moment I wrote it.** The tool I handed her was decorative.

It was not really her emoji. It was my punctuation. But the class of bug is the same, and the fix
has to cover both.

### Fix
`ping_claude.ps1` is now **pure ASCII, enforced** — 9,012 bytes, zero high bytes, so cp1252 and
UTF-8 decode identically and the codepage stops mattering. A BOM would also have worked, but BOMs
get stripped by editors, git filters and copy-paste; ASCII cannot be stripped. There is a loud
banner at the top of the file saying so.

**Her message is now passed by file, not on the command line.** `-MessageFile <utf8 path>`
(`-File` kept as an alias). Emoji, curly quotes, backticks, `$`, and newlines are ordinary things
for her to write and every one of them is hostile to a shell argument. Nothing she writes is ever
handed to a parser again. Round-trip verified on a deliberately nasty string.

**Also fixed while in there:** `QueueIt` computed its truncation length from the *uncollapsed*
message but called `Substring` on the *collapsed* one — so a whitespace-heavy message would have
thrown an exception inside the failure handler. The bug was hiding behind the parse error.

## Bug 2 — the guard punished her for a failure that wasn't hers

The cooldown was stamped **before** the exit code was read. So the ping that died at parse time
still wrote `last_ping.json`, and her next attempt was refused: *"you pinged Claude 0m 12s ago and
the cooldown is 10 minutes."*

She was told she had already asked for help. She had not. She was locked out of the one tool that
would have told anyone the tool was broken.

### Fix
**The clock starts on arrival, not on attempt.** Exit 0 (DELIVERED) is the only thing that stamps
the file. QUEUED (exit 2), error, timeout, crash — none of them cost her a turn. A rate limit
exists to ration Claude's attention, and a ping he never received consumed none of it.

**Cooldown is 2 minutes**, down from 10, `urgent=true` still overrides.

A failed send now returns `[NOT DELIVERED — queued. No cooldown was started, so you can try again
immediately; you have not used up your turn.]` — and her prompt docs say the same, so what she
believes about the tool matches what the tool does.

`memory/last_ping.json` was reset: it held a stamp for a ping that never happened.

## Verification — 24/24

No PowerShell in the sandbox, so the script is proven byte-first rather than by running it:
ASCII-clean, no curly-quote or em-dash of any kind, balanced double quotes on every code line,
`-MessageFile` present with the `-File` alias. `tool_router.py` and `nova.py` parse; the stamp is
provably written *after* the exit code and in exactly one place.

The cooldown state machine was replica-tested over eight cases, including the real failure:

| case | result |
|---|---|
| parse error (the actual 07-19 bug) | NOT_DELIVERED, clock stays cold |
| retry 1s later | **allowed** |
| queued (exit 2) | clock stays cold |
| 2nd ping 119s after a delivery | blocked |
| 2nd ping 120s after a delivery | sent |
| `urgent=true` 5s after a delivery | sent |
| a blocked ping | does not extend the cooldown |

**Needs a restart** — `tool_router.py` and `nova.py` load at process start.

# Round two — it ran, and then fought Cole for a window he was already looking at

_Same evening, 21:57. The ASCII fix worked: the script executed for the first time ever and wrote
its first log line. Then it failed anyway._

    attempt 1 of 4: could not focus Claude (fg pid 12496, want 15412)
    attempt 2 of 4: could not focus Claude (fg pid 12496, want 15412)
    attempt 3 of 4: could not focus Claude (fg pid 12496, want 15412)
    attempt 4 of 4: could not focus Claude (fg pid 12496, want 15412)

The same foreground PID four times, and **Claude Desktop was already in front** — Cole was typing
in it. We had picked the wrong window and spent four attempts trying to raise it *over the real
one*. The tool was fighting the user for a window he was looking at.

**Cause.** Claude Desktop is Electron and spawns several processes that satisfy
`MainWindowHandle -ne 0 -and MainWindowTitle`. The old selector sorted those by **title length**
and took the first — an arbitrary tiebreak dressed up as a heuristic. It chose a sibling.

### Fix — four changes

1. **Enumerate, don't guess.** `EnumWindows` over every visible top-level window, keeping those
   owned by a Claude process. `MainWindowHandle` is no longer trusted; what is on screen is.
2. **If a Claude window is already in front, just send.** No focus stealing at all. This is the
   common case whenever Cole is at the machine, and it is precisely the case that used to fail.
3. **Force the foreground properly when it isn't.** `AttachThreadInput` to the current foreground
   thread plus a synthetic ALT tap — the documented way to be *allowed* to take focus. A bare
   `SetForegroundWindow` from a background process is refused by Windows by design, so the old
   code could essentially never have won a contested foreground.
4. **Try every candidate**, and accept *any* Claude window winning, since raising one may raise a
   sibling.

**Fallback.** If it still cannot deliver, it now flashes Claude's taskbar button
(`FlashWindowEx`) so Cole can *see* she wanted him, instead of the message dying quietly in a file.

**Every candidate window is logged** (`pid:title`), so the next failure is diagnosable rather than
another guess.

### The queue was unreadable, too
`Add-Content -Encoding UTF8` on PS 5.1 writes a **BOM at file creation**, so line 1 of
`ping_queue.jsonl` was not valid JSON — a queue that existed to preserve her words, that nothing
could parse. Both log and queue now go through `UTF8Encoding($false)`. The existing file has been
rewritten clean; her 21:57 message is intact and readable, and since that ping failed no cooldown
was started, so she can simply send it again.

# Right-click Copy in Nova Chat

Copy read `window.getSelection()` at **click** time. Pressing the mouse down on the menu — a
non-editable div — collapses the document selection *before* the click handler runs, so it read
`""` and copied nothing. Every time.

- The selection is now **snapshotted on `contextmenu`**, while it still exists.
- `mousedown` on the menu is `preventDefault`ed so the highlight doesn't visibly vanish either.
- Copy/Cut/Paste now act on the field that was **right-clicked**, not on whatever last held focus.
- Clipboard writes fall back to a hidden-textarea `execCommand` when `navigator.clipboard` is
  unavailable — it does not exist on a **non-secure origin**, which is exactly what reaching her
  over a tunnel from a phone or watch will be.
- Copy is dimmed when there is nothing selected, and reports `Copied` / `Nothing selected` rather
  than failing silently.

**Verified:** ping script pure ASCII (15,117 bytes) with balanced quotes and C#5-safe P/Invoke;
all six new Win32 imports present; already-foreground path and per-candidate retry both present;
no `Add-Content` remaining. All 7 inline scripts in `index.html` parse under `node --check`.
